# Методика клонирования demo.mp4 (как пользоваться циклом)

Цель: довести `index.html` до точного клона геймплейной фазы `demo.mp4`, судя **по реальному рендеру**, а не на глаз (причина прошлых провалов — ТЗ §8).

## Цикл за один проход

```
эталон  →  рендер  →  метрики+глаз  →  правка CFG  →  повтор
refs/      out/        out/cmp_*.png    index.html
```

### 1. Корпус эталонов (один раз)
```
bash tools/extract.sh          # demo.mp4 -> refs/frames/*, refs/contact.png
```
Фазы размечены в `refs/phases.md`. Целевая фаза — **B (геймплей)**. Канонические мишени — `refs/keyframes/kf_{establish,hill,spread,pad}.png`.

### 2. Рендер (детерминированный)
```
node tools/shoot.mjs --seed 7 --suffix _after
# свип значения БЕЗ правки файла:
node tools/shoot.mjs --seed 7 --suffix _try --q "exposure=0.6&groundColor=0x504c5e"
```
- index.html в `?test=1&seed=N`: авто-старт, без звука, seeded-RNG, фикс-шаг через `window.__sim`.
- Любой ключ из `CFG` (вверху index.html) переопределяется через `--q "key=val&..."` или `window.__cfg`.
- Выход: `out/shot_<beat><suffix>.png`, портрет 720×1280.

### 3. Сравнение
```
python tools/compare.py refs/keyframes/kf_hill.png out/shot_hill_after.png --label hill
```
- Кропает реф к игре (убирает рекламный баннер/кнопки), метрики по палитре/яркости/золоту.
- Печатает scorecard, пишет `out/cmp_<label>.png` (REF|SHOT) и строку в `out/report.md`.
- **SCORE ↓** — суммарная дельта (ground+gold RGB, luma, доля золота). Меньше = ближе.
- Финальный судья — глаз на `out/cmp_*.png`: метрика не ловит камеру/композицию.

### 4. Правка
Нашёл значение свипом — запекай в дефолты `CFG` вверху `index.html`. Коммить дельту.
Правило: меняй по одной группе (палитра / камера / ворота), сверяй SCORE, не регрессируй.

## Следующие итерации (по убыванию разрыва, видно на cmp_*.png)

1. **Камера/композиция** — реф площе и шире (дозер мелкий, длинное поле). Крутить `CFG.camHeight/camBack/fov/lookAhead`.
2. **Ворота** — реф: крупный красный шеврон ×N (и белая завеса ×10), не сине-голубой. Перекрасить `gateTex`/добавить тип-шеврон; завязать на `CFG.gateCurtain/gateGlow`.
3. **Сценарий бет spread/pad** — наши монеты ссыпаются в пад → доля золота не совпадает с рефом. Подогнать `SCEN` в `tools/shoot.mjs` под тайминг ролика.
4. **Пады** — рампа + «UPGRADE» как в рефе.

## Калибровка камеры рефа (выученное трудным путём)

Итоговая камера (зафиксирована в `CFG`): **yaw 49.4°, pitch 54.5°, f 1600px (fov 43.6° на наш вьюпорт 1280)** — фит по осевым линиям пользователя на f_0620, rms 0.65°; вертикали сходятся в 1.6°.

### Что НЕ работает

- **Снимать углы «на глаз» по пикселям кадра** — три провала подряд (+7° → −13° → круговое сравнение собственной линии с собой). Любая координата, прочитанная глазом с уменьшенной картинки — подозрительна.
- **Сравнивать наклон линии между двумя скриншотами при разной композиции.** Наклон проекции мировой прямой зависит от её места в кадре (сходимость к VP). Две линии одного мирового направления пересекаются даже в пределах ОДНОГО кадра (см. `out/perspective_proof.png`). Сравнивать можно только при совмещённой композиции или через VP.
- **Авто-извлечение семейства вертикалей из 2 рёбер одного столба** — слабое звено: ошибка yaw на 11° при «успешной» калибровке. Мало линий = мусорная VP, даже если решение проходит санити-гейты.
- **Медиана по смеси сегментов ролика.** Рекламный монтаж = несколько камер (ранний широкий ~45°, поздний ~54°). Усреднение по всем кадрам даёт камеру, которой нет ни в одном кадре.

### Что работает

- **Пользователь рисует цветные линии на кадре → извлечение штрихов диффом против оригинала + цветом → PCA-фит.** Глаз человека размечает семейства лучше любых моих масок; координаты при этом точные (не «на глаз»).
- **Совместный фит (yaw, pitch, f) по всем штрихам сразу**: направление штриха в точке должно смотреть на VP семейства; минимизация угловой ошибки grid-search'ем. 13 линий → rms 0.65°.
- **Перекрёстные проверки**: вертикали (не участвовали в фите) предсказываются той же камерой; эллипсы монет (круг→эллипс, отношение осей = sin(pitch)) — независимый pitch; «клин» вертикалей от оснований = веер к VP ниже кадра (601px под краем) — норма, не баг.
- **Самопроверка пайплайна на своём рендере** (камера известна) до прикосновения к рефу: `python tools/_calib.py selftest`.
- **Principal point рефа = центр игровой зоны (720×1080, pp y=540), не центр видео (1280).** Рекламный баннер снизу — отдельная зона. При переносе в наш полноэкранный вьюпорт (pp 640) поправка pitch: −atan(100/f) ≈ 3.6°.
- **Перенос в нашу follow-камеру**: `camYaw = yaw`; `pitch = atan(camHeight/(camBack+lookAhead))`; `fov` — на полную высоту вьюпорта. Перенос точен по построению — отдельная «проверка калибратором своего рендера» не нужна (и недостоверна, пока в сцене нет длинных полос-источников LONG-линий).

Инструменты: `tools/_calib.py` (selftest / frame / batch), разметка пользователя — `out/ref_vs_ours_anchor_ed.png` (red=LAT, blue=LONG, салатовый=VERT).
